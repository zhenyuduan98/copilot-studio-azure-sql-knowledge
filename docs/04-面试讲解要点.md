# 面试讲解要点

帮你把「把 Azure SQL 接成 Copilot Studio 知识库」这个项目讲清楚、讲出亮点、应对追问。建议讲解 5–8 分钟，配合现场或录屏演示。

---

## 一、一句话电梯陈述

> 「我把一个 Azure SQL 数据库直接挂成 Copilot Studio 智能体的知识源，做了个房地产房源助手：经纪人用大白话提问——比如『有没有三居室？』『给我看特拉华州的房子』——智能体就能从数据库里查出对应房源作答，全程不写一行 SQL。」

---

## 二、讲解主线（建议顺序）

1. **痛点**：企业核心数据都在数据库里，业务人员查数据要写 SQL 或依赖 BI，有门槛、不灵活。
2. **方案**：把 Azure SQL 直接挂成智能体知识源，自然语言直达数据库。
3. **三个关键设计**（重点，见下「技术亮点」）：
   - 用 **Service Principal** 共享鉴权 + **只读最小权限**；
   - 用 **同义词 / 描述 / 数据词汇表** 把业务话翻译成库里的字段和代码；
   - 清楚它的**适用边界与限制**。
4. **现场演示**：问三类问题（按条件筛选 / 词汇表映射 / 顺外键查经纪人）。
5. **价值与延展**：任何「让业务用自然语言查结构化数据」的场景都能复用。

---

## 三、技术亮点（面试官最想听的）

### 亮点 1：Service Principal 共享鉴权 + 最小权限
- **问题**：如果用每个人自己的账号连 SQL，要给每个用户单独建连接、单独授权，根本没法规模化。
- **解法**：注册一个 **Service Principal（Entra ID 应用）**，连接一次配置、**所有用户共享**；
- **安全**：在数据库侧把这个服务主体建成**只读外部用户（db_datareader）**——智能体只能读、不能改不能删。
- **体现**：既懂**企业级共享鉴权**，又有**最小权限**的安全意识。

### 亮点 2：三层语义增强让模型「听懂业务话」
- 列名是 `listing_id`、状态是 `DE`，但用户说「MLS 编号」「特拉华州」。
- **同义词** 给列起别名、**描述** 补充列含义、**数据词汇表** 把值（DE↔Delaware）对齐。
- **体现**：理解到「接上数据库」和「真正好用」之间隔着一层**业务语义层**，这是工程细致度的体现。

### 亮点 3：清楚边界与限制（最能体现成熟度）
- 大表有**行数上限**（约 6k 量级），超了结果会过时/随机；
- 多行返回**完整性不保证**（查 5 条可能只回 2 条）；
- 所以我会说：**甜区是中小数据的自然语言查询和快速交付**，大表/精确聚合要做视图预筛选或回退 SQL。
- **体现**：不是只会「把 demo 跑通」，而是知道**什么时候不该用、怎么规避**。

### 亮点 4：低代码快速交付
- 整套不写代码：建库、建表、配鉴权、挂知识源、配语义——属于 **低代码 + AI** 的企业落地范式，可快速复制到其它数据集。

## 四、可能被追问的问题与答法

**Q：为什么用 Service Principal，不用普通账号？**
A：服务主体的连接可以被所有智能体用户共享，不用每个人单独建连接和授权；而且能在数据库侧只授予只读权限，安全可控。普通账号既难共享又容易过度授权。

**Q：怎么保证安全？智能体会不会改坏数据？**
A：不会。数据库侧给服务主体建的是 `db_datareader` 只读外部用户，权限上就只能查询。加上网络层「只允许 Azure 服务 + 指定 IP」访问，双重收口。

**Q：列名和用户说法不一致怎么办？比如用户说『MLS 编号』？**
A：用同义词把「MLS 编号」绑到 `listing_id` 列；类似地用数据词汇表把「Delaware」映射到州代码 `DE`，用列描述补充每列的含义和单位。这三层让模型能把自然语言对齐到真实字段。

**Q：数据量很大也能用吗？**
A：要小心。智能体拉取有行数上限（约 6k 量级），大表会导致结果不准。生产上我会先做视图或预筛选把数据压到上限内，精确聚合场景则回退到真正的 SQL 查询。

**Q：它是把整张表喂给大模型吗？数据隐私如何？**
A：它是按问题去查询数据库的相关数据，不是全量灌给模型；连接走只读服务主体，权限和网络都做了限制。具体数据驻留与合规以租户/环境策略为准。

**Q：和直接做个 BI 报表比，优势在哪？**
A：BI 是预定义好的固定视图，换个问法就要重新做报表；这套是**自然语言即查询**，灵活、零门槛，适合长尾、临时的提问场景。两者可以互补。

**Q：Copilot Studio 连 SQL 和「连文档知识库」有什么区别？**
A：文档知识库是非结构化检索（RAG）；连 SQL 是面向**结构化数据**，能做按字段筛选、跨表（外键）关联这类精确查询，更适合数据库里的业务数据。

## 五、演示脚本（约 1 分钟）

依次问三类问题，覆盖三种能力：

1. **按条件筛选**：「有没有 4 间卧室、50 万美元以内的房子？」→ 展示按 `bed`、`price` 过滤。
2. **数据词汇表映射**：「给我看特拉华州（Delaware）的房源。」→ 展示 Delaware→`DE` 的映射。
3. **顺外键关联**：「MLS 编号 7 的房子是谁负责的？电话多少？」→ 展示从房源表顺 `agent_id` 外键查到经纪人联系方式。

收尾一句：「同样一套方法，把表和语义配置换掉，就能套到订单、库存、客户任何业务数据上。」

---

## 六、容易加分的细节

- **db_datareader 只读**：主动强调最小权限，安全意识立刻凸显。
- **Client Secret 只显示一次**：提一句「创建后要立即保存」，显示你真动手做过。
- **行数上限**：能主动讲限制和规避手段，比只会搭建更显成熟。
- **网络两道口子**：「允许 Azure 服务访问」给 Copilot Studio、防火墙 IP 给本地 SSMS——能讲清这两者的区别是加分项。
