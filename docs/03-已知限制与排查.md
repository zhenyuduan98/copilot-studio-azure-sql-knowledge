# 已知限制与排查

这套方案很适合做演示和中小规模数据查询，但用于生产前，下面几个限制必须先了解和实测。以下整理自原文评论区里真实用户反馈的问题。

---

## 1. 大表有行数上限（最重要）

- **现象**：有用户的表约 **60 万行**，智能体返回的结果非常**过时/随机**，明显不对。
- **原因**：智能体从 SQL 拉取的数据有**行数上限**（反馈中提到约 **6000 行** 量级）。超出后它只看到部分数据，自然答不准。
- **影响**：不能指望它在一张大表上做「全量精确统计/检索」。
- **建议**：
  - 演示和中小表（千级别）场景没问题；
  - 大表先**做视图或预筛选**，把数据量压到上限以内；
  - 涉及精确聚合（求和、计数、排名）时尤其要小心，务必抽样核对。

---

## 2. 多行结果可能返回不全

- **现象**：有用户查「**最近 5 天**」的数据，库里明明有 5 条，智能体只返回了 **2 条**。
- **原因**：多行检索的**完整性/可靠性**不稳定，不保证「应有尽有」。
- **建议**：
  - 对「列出所有符合条件的记录」这类需求要**实测验证**；
  - 关键业务别直接依赖它的列表结果，必要时回退到真正的 SQL 查询或报表。

---

## 3. 用了 Service Principal，用户仍被要求输入凭据

- **现象**：即使配置了服务主体、共享了连接，部分终端用户仍被提示输入凭据，并报错 **「Unable to provision connection（无法预配连接）」**。
- **可能方向**：
  - 确认连接确实以**服务主体**方式建立并已**共享**给使用者；
  - 核对第 2 步网络里 **「允许 Azure 服务访问此服务器」** 已勾选；
  - 核对数据库侧外部用户名称与应用注册**完全一致**、且授予了 `db_datareader`；
  - 在发布/共享智能体时检查连接引用是否正确。

---

## 4. 排查清单（搭建出问题时按序检查）

| 症状 | 优先检查 |
|------|----------|
| 完全连不上 / 报无法预配连接 | 网络「允许 Azure 服务访问」是否勾选；Tenant/Client ID/Secret 是否填对；Secret 是否过期 |
| 连上了但查不到数据 | 外部用户名是否与应用注册完全一致；是否授予 `db_datareader`；表是否被勾选进知识源 |
| 换个说法就查不到 | 同义词 / 列描述 / 数据词汇表是否配置（见 `02-知识增强配置.md`）|
| 大表结果过时/随机 | 命中行数上限，做视图或预筛选压缩数据量 |
| 列表结果缺条目 | 多行返回不可靠，实测核对，关键场景回退 SQL |

---

## 5. 一句话总结（面试可用）

> 「这套方案的甜区是**中小规模结构化数据的自然语言查询**和**快速演示**；一旦表很大或需要精确的多行/聚合结果，就要注意**行数上限**和**返回完整性**这两个限制，生产前必须实测，必要时用视图预筛选或回退到传统 SQL。」

了解并能讲清楚这些限制，往往比只会搭建更能体现工程成熟度。
