# 知识增强配置（同义词 / 列描述 / 数据词汇表）

把 Azure SQL 表接进来只是第一步。表里的列名是 `listing_id`、州是两位代码 `DE`，而用户嘴里说的是「MLS 编号」「特拉华州」。这三样配置负责把**业务语言**翻译成**数据库语言**，是提问能否被准确理解的关键。

| 机制 | 作用对象 | 一句话 |
|------|----------|--------|
| 同义词 Synonyms | **列名** | 给列起别名 |
| 描述 Descriptions | **列含义** | 给列补充上下文说明 |
| 数据词汇表 Glossary | **数据值** | 给具体取值补充说明/映射 |

配置入口：知识源 → 三个点 → 编辑，进入对应表/列的设置。

---

## 1. 同义词（Synonyms）—— 给列起别名

让用户用不同叫法都能命中同一列。

**示例**：`listing_id` 列添加同义词：
- MLS 编号（MLS Number）
- Property ID
- Reference Number

配置后，用户问「**MLS 编号 7 的房子**」时，智能体知道「MLS 编号」= `listing_id`，于是去查 `listing_id = 7`。

**建议加同义词的列**：
| 列 | 可加的同义词 |
|----|-------------|
| `listing_id` | MLS 编号、房源编号、Property ID、Reference Number |
| `bed` | 卧室、几居室、bedrooms |
| `bath` | 卫生间、浴室、洗手间、bathrooms |
| `price` | 价格、报价、挂牌价、售价 |
| `square_feet` | 面积、建筑面积、平方英尺 |

---

## 2. 列描述（Descriptions）—— 给列补充语义

为用到的每一列写清楚「这列是什么、单位是什么、取值规律」，帮模型更准地理解和筛选。原文建议**给所有会被查询的列都写描述**。

**示例**：
- `bed`：房源的卧室数量，整数，例如 3 表示三居室。
- `bath`：卫生间数量，允许半间，例如 2.5 表示两个全卫加一个半卫。
- `price`：挂牌价，单位美元；用户问「50 万以内」对应 `price <= 500000`。
- `state`：美国州的两位字母代码（如 NY、DE、PA）；用户常说全称，需配合数据词汇表。
- `listed_date`：挂牌日期；用户问「最近挂牌」对应按此列倒序。

---

## 3. 数据词汇表（Data Glossary）—— 给「值」补充说明

同义词管的是**列名**，数据词汇表管的是**列里的具体值**。

**经典示例（州代码）**：数据库 `state` 列存的是两位代码 `DE`，但用户会问「**给我看特拉华州（Delaware）的房子**」。把每个州的「全称 ↔ 代码」加进词汇表：

| 全称 | 代码 |
|------|------|
| Delaware | DE |
| New York | NY |
| Pennsylvania | PA |
| Massachusetts | MA |
| Virginia | VA |
| Maryland | MD |

配置后，「Delaware」会被映射到 `DE`，从而正确筛选。

---

## 为什么这三层很重要（面试可讲）

- 没有它们，模型只能在**字面**上匹配列名和值，用户稍微换个说法就查不到；
- 有了它们，相当于给数据库加了一层**业务语义层**，把「人话」和「库里的字段/代码」对齐；
- 这是「**接上数据库**」和「**真正好用**」之间的差距所在——也是这套方案里最体现工程细致度的部分。
